11. Векторы версий (Version vectors). Векторы версий с записями для каждого клиента/записями для каждого сервера. Векторы версий “с точками” (Dotted Version Vectors).

"Dotted version vectors" - это вариация векторных часов, где каждая временная метка представлена парой чисел: число и "точка". Когда узел видит событие, он увеличивает свою временную метку, добавляя ей "точку". Это помогает разрешить конфликты, когда два события имеют одинаковое время. Путем добавления "точек" система может уточнить порядок между ними.

Векторы версий

Векторы версий лучше подходят для приложений, которые хранят данные, где каждый элемент данных помечен вектором версии. Таким образом, данные потенциально могут обновляться в нескольких частях системы одновременно (например, при наличии сетевого разрыва), так что векторы версий из результирующих элементов данных могут помочь нам идентифицировать те элементы, которые могут быть согласованы автоматически, и те, которые требуют разрешения конфликтов.
Векторы версий поддерживают состояние, идентичное состоянию векторных часов, содержащих по одной целочисленной записи на узел в системе. Правила обновления немного отличаются: разные узлы могут получать как локальные обновления (например, запись, выполненная на сервере), так и синхронизироваться с другим узлом (например, при восстановлении из сетевого раздела).

Идеи

  • Изначально все элементы всех векторов установлены на ноль.
  • Каждый раз, когда узел испытывает событие локального обновления, он увеличивает свой собственный счетчик в векторе на единицу.
  • Каждый раз, когда два узла a и b синхронизируются, они оба устанавливают элементы в своем векторе на максимальное значение элементов в обоих векторах Va[x]=Vb[x]=max(Va[x],Vb[x])V_{a[x]} = V_{b[x]} = max(V_{a[x]}, V_{b[x]})Va[x]​=Vb[x]​=max(Va[x]​,Vb[x]​)﻿. После синхронизации оба узла будут имеют одинаковые векторы. Кроме того, в зависимости от того, были ли исходные векторы причинно связаны или нет, один из связанных элементов заменит другой или будет выполнена некоторая логика разрешения конфликтов для сохранения одной записи, связанной с новым вектором.

Векторы версий в основном полезны в системах, которые действуют как хранилища данных, поэтому узлы в системе будут принадлежать к двум основным категориям: серверные узлы (узлы-реплики), которые хранят данные и получают операции чтения/записи, и клиентские узлы, которые считывают данные с узлов-реплик и отправляют инструкции по обновлению. В во многих случаях клиенты могут даже не быть частью наших систем, например, в сценариях, когда наша система получает операции непосредственно из веб-браузеров клиентов. В результате было бы лучше избежать значительного объема логики и накладных расходов на хранение в клиентских узлах. Механизм вектора версий позволяет это следующим образом:

  • для каждого узла (как реплики/сервера, так и клиентских узлов) поддерживается одна запись. Однако клиентские узлы могут не иметь состояния, что означает, что они не хранят векторы версий. Вместо этого они получают вектор версии как часть каждой операции чтения и они предоставляют этот вектор версии при выполнении операции обновления обратно соответствующему узлу реплики. Этот вектор называется контекстом, и он используется узлом реплики для соответствующего обновления своего вектора версии.
  • Пояснение

    у нас есть 3 клиентских узла и 2 серверных. V,W,X,Y - какие-то наши документы. Изначально мы клиент просто отправляет документ на сервер. Но потом ему нужно также указывать вектор-контекст: версию, которую он обновляет

Для простоты в примере предполагается, что существует только один элемент, поэтому все клиенты работают с одним и тем же элементом. Его можно легко расширить, чтобы охватить случаи с несколькими элементами, где у каждого элемента будет отдельный вектор версии, и клиенты также должны будут предоставить идентификатор для элемента, к которому будет осуществлен доступ. Каждая операция чтения возвращает текущее значение с соответствующим вектором версии. Каждая операция записи имеет 3 аргумента: первый - это вектор версии, который задается в качестве контекста, второй - идентификатор клиентского узла и последний - значение, подлежащее записи. Узел реплики отвечает за увеличение соответствующей записи в векторе (в зависимости от идентификатора клиента) и затем сохраняем новое значение. Обратите внимание, что это новое значение может либо сохраняться вместе с другими значениями, которые были записаны одновременно, либо перезаписывать значения, которые причинно предшествуют ему. В первом случае при последующем чтении будет возвращено несколько значений, и следующие операции записи согласуют их, сохраняя одно значение. Например, в нашем примере ниже, если узел D выполнил чтение из узла B (который вернет оба значения V и W и их векторы версий), а затем попытался записать значение Z, затем обновление будет иметь вид PUT({(C,1), (D,1)}, D, Z). Узел реплики вычислит новый вектор версии {(C,1), (D,2)} и определит, что он заменяет оба существующих вектора версий {(C,1)} и {(D,1)}, поэтому он заменит оба значения V и W на Z.

Проблемы

Подход, включающий записи в векторные часы для всех клиентских узлов, безопасен и позволяет успешно идентифицировать, когда одновременно были записаны 2 разные версии или одна из них причинно предшествует другой и может быть отброшена. Однако его основным ограничением является плохая масштабируемость. В распределенных системах, которые используются в качестве распределенных хранилищ данных, количество клиентов, как правило, намного превышает количество серверных узлов на два или три порядка. Например, во многих случаях каждый элемент реплицируется на 3 разных серверах, в то время как к этому элементу могут обращаться тысячи клиентов. Обратите внимание, что даже в тех случаях, когда клиентами систем являются несколько серверов приложений, если каждый сервер выполняет операции одновременно из нескольких потоков необходимо поддерживать отдельную запись в векторных часах для каждого из них. В результате такой подход требует значительного объема памяти.

Путь решения

В идеале мы хотели бы, чтобы размер векторных часов масштабировался в зависимости от количества серверных узлов, а не от количества клиентов. Не могли бы мы, возможно, удалить клиентские записи из векторных часов и позволить серверам увеличивать свои собственные записи при выполнении обновлений от имени клиентов? К сожалению, нет. Если бы мы это сделали, система не смогла бы обнаружить, что некоторые операции выполнялись одновременно, таким образом, отбрасывались значения, которые должны были быть сохранены.

Как вы можете видеть, первые операции записи, выполняемые клиентскими узлами C и D, выполняются одновременно. Однако серверный узел B не смог бы идентифицировать это. Узел B рассматривал бы вектор версии второго обновления {(B,2)} как заменяющий {(B,1)}, таким образом отбрасывая значение V и заменяя его значением W.

Точечные векторы версий

Существует методика, позволяющая успешно идентифицировать параллельные версий, в то же время позволяя векторам версий масштабироваться в зависимости от количества серверов, называемых точечными векторами версий. У этого метода есть особенность, позволяющая достичь этого: каждая запись в векторе больше не является одним числом, а парой чисел. Это может кодировать последовательность чисел, которые не являются полностью последовательными, но могут содержать один пробел. Пара (n1,n2) представляет все числа от 1 до n1 плюс число n2.

Например, пара (4,7) представляет последовательность [1,2,3,4,7]. Обратите внимание, что вторая номер необязателен, и некоторые записи все равно могут содержать один номер. Это можно использовать для отслеживания параллелизма между несколькими версиями.

Порядок между двумя версиями теперь определяется в терминах отношения contains в соответствующих последовательностях. Итак, для векторов v1, v2 соотношение v1 ≤ v2 выполняется, если последовательность, представленная v1, является подмножеством последовательности, представленной v2. Более конкретно:

  • (m) ≤ (m0), если m ≤ m0
  • (m) ≤ (m0, n0) , если m ≤ m0 ∨ m = m0 + 1 = n0
  • (m, n) ≤ (m0), если n ≤ m0
  • (m, n) ≤ (m0, n0)) , если n ≤ m0 ∨ (m ≤ m0 ∧ n = n0)

Правило обновления, выполняемое каждым узлом-репликой при получении операции записи, также немного отличается. Для всех индексов, кроме тех, которые принадлежит узлу-реплике, узел использует значение (m), где m – максимальное число среди доступных в предоставленных векторах версий в контексте. Для индекса, соответствующего узлу реплики, узел использует пару (m, n+1), где m – максимальное число среди доступных в предоставленных векторах версий в контексте, а n - максимальное число среди векторов версий, присутствующих в узле реплики (по сути, значение его логических часов).

Как вы можете видеть, операции записи клиентскими узлами C
и D заканчиваются получением векторов версий {(B,0,1)} и {(B,0,2)}, и они успешно идентифицируются как параллельные, поскольку {(B,0,1)} не меньше {(B,0,2)} и {(B,0,2)} не меньше {(B,0,1)}.
